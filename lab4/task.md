## 2. Теоретические сведения

### 2.1 Модель ISO/OSI

Модель OSI описывает взаимодействие сетевых приложений через уровни:

|№|Уровень|Назначение|Использование в проекте|
|---|---|---|---|
|7|**Application (Прикладной)**|Логика приложения|обработка сообщений (`MSG_TEXT`, `MSG_PRIVATE`, `MSG_AUTH`)|
|6|**Presentation (Представления)**|Формат данных, сериализация|структура `Message`, упаковка/распаковка|
|5|**Session (Сеансовый)**|Управление сессией|аутентификация, состояние клиента|
|4|**Transport (Транспортный)**|Передача данных|TCP (`send`, `recv`)|
|3|**Network (Сетевой)**|IP-адресация|используется через сокеты (IP)|
|2|**Data Link (Канальный)**|передача кадров|Ethernet / Wi-Fi (не реализуется)|
|1|**Physical (Физический)**|передача битов|кабель / сигнал (не реализуется)|

---

### 2.2. Аутентификация

Аутентификация — процесс идентификации пользователя.

В данной работе:

- клиент обязан отправить никнейм
- сервер запрещает работу без авторизации
- никнеймы уникальны

---

### 2.3. Личные сообщения

Приватный канал связи между двумя клиентами:


```cpp
ClientA → Server → ClientB
```

Сервер выполняет маршрутизацию сообщений.

## 3. Постановка задачи

На основе ЛР3 необходимо:

### 3.1. Расширить протокол

Добавить новые типы сообщений:

```cpp
enum {
    MSG_HELLO = 1,
    MSG_WELCOME = 2,
    MSG_TEXT = 3,
    MSG_PING = 4,
    MSG_PONG = 5,
    MSG_BYE = 6,

    MSG_AUTH = 7,        // аутентификация
    MSG_PRIVATE = 8,     // личное сообщение
    MSG_ERROR = 9,       // ошибка
    MSG_SERVER_INFO = 10 // системные сообщения
};
```

---

### 3.2. Модифицировать сервер

#### 1. Аутентификация

1. После подключения сервер ожидает **только `MSG_AUTH`**
2. Пока клиент не авторизован:
    - все остальные сообщения игнорируются
3. Проверки:
    - ник не пустой
    - ник уникален
4. При ошибке:
    - отправить `MSG_ERROR`
    - отключить клиента

#### 2. Хранение клиентов

```cpp
typedef struct {
    int sock;
    char nickname[32];
    int authenticated;
} Client;
```

Сервер хранит список клиентов с никнеймами.

#### 3. Обработка сообщений

После аутентификации:

- `MSG_TEXT`  
    → широковещательная рассылка  
    → формат:
    
    ```cpp
    [nickname]: сообщение
    ```
    
- `MSG_PRIVATE`  
    → отправка только указанному пользователю  
    → формат payload:
    
    ```cpp
    target_nick:message
    ```
    
- `MSG_BYE`  
    → удаление клиента

#### 4. Личные сообщения

Сервер должен:

1. Найти получателя по нику
2. Если найден:
    
    ```cpp
    [PRIVATE][sender]: сообщение
    ```
    
3. Если нет:  
    отправить `MSG_ERROR`

#### 5. Системные сообщения

При подключении:

```cpp
User [nickname] connected
```

При отключении:

```cpp
User [nickname] disconnected
```

---

### 3.3. Визуализация модели OSI (обязательное требование)

Сервер должен выводить обработку сообщений по уровням:

Пример приёма сообщения
```cpp
[Layer 4 - Transport] recv()
[Layer 6 - Presentation] deserialize Message
[Layer 5 - Session] client authenticated
[Layer 7 - Application] handle MSG_TEXT
```

Пример отправки
```cpp
[Layer 7 - Application] prepare response
[Layer 6 - Presentation] serialize Message
[Layer 4 - Transport] send()
```

#### Требования

1. Вывод обязателен для:
    - всех входящих сообщений
    - всех исходящих сообщений
2. Уровни должны быть явно подписаны:
    
    ```cpp
    Layer 4, Layer 5, Layer 6, Layer 7
    ```

---

### 3.4. Модифицировать клиент

1. При запуске:
    - запросить никнейм
    - отправить `MSG_AUTH`
2. Поддержка команд:
```cpp
/w <nick> <message>   → MSG_PRIVATE  
/quit                 → MSG_BYE  
/ping                 → MSG_PING
```

3. Отображение:
```cpp
[user]: message  
[PRIVATE][user]: message  
[SERVER]: message
```

---

## 4. Алгоритм работы сервера

1. `socket()`
2. `bind()`
3. `listen()`
4. `accept()`
5. Ожидание `MSG_AUTH`
6. Проверка никнейма
7. Добавление клиента в список
8. Цикл обработки:
    - TEXT → broadcast
    - PRIVATE → отправка одному
    - BYE → отключение
9. При разрыве:
    - удаление клиента

---

## 5. Алгоритм работы клиента

1. `socket()`
2. `connect()`
3. ввод никнейма
4. отправка `MSG_AUTH`
5. получение ответа
6. цикл:
    - ввод пользователя
    - отправка сообщений
7. поток приёма сообщений

---

## **Примечание**

В рамках данной лабораторной работы предполагается, что начальный обмен служебными сообщениями, введённый ранее, уже функционирует и не требует изменений. Речь идёт о взаимодействии, происходящем сразу после установления TCP-соединения, которое обеспечивает первичную синхронизацию клиента и сервера.

Данный этап рассматривается как предварительный и в текущей работе не модифицируется. Дальнейшая логика (включая идентификацию пользователя) выполняется только после завершения указанного взаимодействия.
