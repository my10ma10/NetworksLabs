# Лабораторная работа №5

## Расширенный TCP-чат с форматом `MessageEx`, хранением истории в файле, офлайн-сообщениями и логированием по модели TCP/IP

---

## 1. Цель работы

Расширить функциональность Лабораторной работы №4:

- Необходимо заменить модель OSI на модель **TCP/IP**
- Расширить формат сетевого сообщения
- Добавить **временные метки**, **идентификатор отправителя**, **сохранение истории чатов в файл**
- Реализовать механизм **Store & Forward** для личных сообщений, когда адресат временно не подключён

---

## 2. Теоретические сведения

### 2.1 Модель TCP/IP

В отличие от ЛР4, где использовалась визуализация уровней OSI, в ЛР5 необходимо использовать **модель TCP/IP**. В рамках проекта она интерпретируется так:

|Уровень|Назначение|Использование в работе|
|---|---|---|
|Application|прикладная логика|аутентификация, команды, чат, история, офлайн-доставка|
|Transport|транспортная передача|TCP, `send()`, `recv()`|
|Internet|IP-адресация|IP-адрес клиента, маршрутизация через сокеты|
|Network Access|доступ к физической среде|Ethernet/Wi-Fi, показывается только в логах|

---

### 2.2 Расширенный формат сообщения

В ЛР2 использовалась структура `Message` с полями `length`, `type`, `payload`.   
В ЛР5 необходимо перейти к новому формату **`MessageEx`**, который хранит больше служебной информации и делает протокол более наглядным для студентов.

---

### 2.3 Хранение истории в файле

История чата должна сохраняться не только в памяти, но и в файл. Это упрощает реализацию просмотра сообщений и позволяет использовать сервер как источник состояния между подключениями пользователей.

---

### 2.4 Store & Forward

Если получатель личного сообщения отсутствует в сети, сервер не должен терять сообщение. Вместо этого оно сохраняется в очередь офлайн-сообщений и доставляется при следующем успешном подключении адресата.

---

## 3. Постановка задачи

На основе ЛР4 необходимо модернизировать сервер и клиент.

### 3.1 Новый формат сообщения

Необходимо заменить формат `Message` на `MessageEx`:

```cpp
#include <stdint.h>
#include <time.h>

#define MAX_NAME     32
#define MAX_PAYLOAD  256
#define MAX_TIME_STR 32

typedef struct {
    uint32_t length;                 // длина полезной части
    uint8_t  type;                   // тип сообщения
    uint32_t msg_id;                 // уникальный идентификатор сообщения
    char     sender[MAX_NAME];       // ник отправителя
    char     receiver[MAX_NAME];     // ник получателя или "" если используется broadcast
    time_t   timestamp;              // время создания
    char     payload[MAX_PAYLOAD];   // текст / данные команды
} MessageEx;
```

---

### 3.2 Расширить протокол

Добавить новые типы сообщений:

```cpp
enum {
    MSG_HELLO        = 1,
    MSG_WELCOME      = 2,
    MSG_TEXT         = 3,
    MSG_PING         = 4,
    MSG_PONG         = 5,
    MSG_BYE          = 6,

    MSG_AUTH         = 7,
    MSG_PRIVATE      = 8,
    MSG_ERROR        = 9,
    MSG_SERVER_INFO  = 10,
	
	MSG_LIST         = 11,  // список пользователей
    MSG_HISTORY      = 12,  // запрос истории
    MSG_HISTORY_DATA = 13,  // ответ с историей
    MSG_HELP         = 14   // справочная информация
};
```

## 4. Требования к серверу

### 4.1 Базовая логика

Сервер должен сохранить основные возможности предыдущих лабораторных работ:

- работу по TCP;
- многопоточную обработку клиентов;
- постоянные соединения;
- аутентификацию по уникальному нику;
- широковещательные сообщения;
- личные сообщения. 

---

### 4.2 Временные метки и ID отправителя

Каждое сообщение должно содержать:

- `msg_id` — уникальный идентификатор;
- `sender` — ник отправителя;
- `timestamp` — время отправки.

При выводе на клиенте и сервере сообщение должно отображаться с временной меткой, например:

```cpp
[2026-04-07 14:35:20][id=42][Alice]: Hello
```

Для личного сообщения:

```cpp
[2026-04-07 14:36:01][id=43][PRIVATE][Bob -> Alice]: Hi
```

---

## 4.3 Сохранение истории в файл в формате JSON

Сервер обязан сохранять историю сообщений в **JSON-файл**.  
### Формат записи истории

Каждое сообщение должно сохраняться как JSON-объект со следующими полями:

- `msg_id` — уникальный идентификатор сообщения;
- `timestamp` — время отправки;
- `sender` — отправитель;
- `receiver` — получатель или пустая строка;
- `type` — тип сообщения;
- `text` — текст сообщения;
- `delivered` — флаг доставки;
- `is_offline` — признак офлайн-сообщения.

### Пример JSON-записи

Для обычного сообщения:

```cpp
{
  "msg_id": 42,
  "timestamp": 1712493320,
  "sender": "Alice",
  "receiver": "",
  "type": "MSG_TEXT",
  "text": "Hello",
  "delivered": true,
  "is_offline": false
}
```

Для личного сообщения, сохранённого в офлайн-очередь:

```cpp
{
  "msg_id": 43,
  "timestamp": 1712493381,
  "sender": "Bob",
  "receiver": "Alice",
  "type": "MSG_PRIVATE",
  "text": "Hi",
  "delivered": false,
  "is_offline": true
}
```

---

### 4.4 Очередь офлайн-сообщений (Store & Forward)

Если клиент отправляет команду `/w <nick> <message>`, а получатель не подключён:

- сервер не отправляет `MSG_ERROR` сразу;
- сервер сохраняет сообщение в очередь офлайн-сообщений;
- запись о сообщении сохраняется в файл истории с `delivered=false`.

Можно использовать структуру:

```cpp
typedef struct {  
    char sender[32];  
    char receiver[32];  
    char text[256];  
    time_t timestamp;  
    uint32_t msg_id;  
} OfflineMsg;
```

При следующем подключении клиента:

- сервер проверяет наличие офлайн-сообщений;
- отправляет их получателю как `MSG_PRIVATE`;
- в тексте или метаданных должна быть пометка `[OFFLINE]`;
- после успешной отправки сообщение помечается как доставленное.

Пример отображения:

```cpp
[2026-04-07 15:10:00][id=57][OFFLINE][Bob -> Alice]: Не забудь отчёт
```

---

### 4.5 Логирование по модели TCP/IP

В ЛР4 требовалась визуализация OSI; в ЛР5 вместо неё необходимо реализовать **логирование по модели TCP/IP**. 

Логи должны выводиться для всех входящих и исходящих сообщений.

Пример приёма сообщения:

```cpp
[Application] deserialize MessageEx -> MSG_PRIVATE
[Transport] recv() 128 bytes via TCP
[Internet] src=127.0.0.1 dst=127.0.0.1 proto=TCP
[Network Access] frame received via network interface
```

Пример расширенного варианта с демонстрацией инкапсуляции:

```cpp
[Network Access] frame arrived from NIC  
[Internet] simulated IP hdr: src=127.0.0.1 dst=127.0.0.1 proto=6  
[Transport] simulated TCP hdr: src_port=54321 dst_port=8080 seq=100 ack=200  
[Application] deserialize -> MSG_PRIVATE from Bob
```

При отправке:

```cpp
[Application] prepare MSG_HISTORY_DATA  
[Transport] send() via TCP  
[Internet] destination ip = 127.0.0.1  
[Network Access] frame sent to network interface
```

Требования:

- логирование обязательно для входящих сообщений;
- логирование обязательно для исходящих сообщений;
- уровни TCP/IP должны быть явно подписаны;

## 5. Требования к клиенту

Клиент должен сохранить общую схему работы из предыдущих лабораторных работ:

- подключение к серверу;
- начальный служебный обмен;
- отправка `MSG_AUTH`;
- отдельный поток приёма сообщений;
- обработка пользовательских команд.

Поддерживаемые команды:

```cpp
/help                 - вывести список доступных команд  
/list                 - запросить список подключённых пользователей  
/history              - запросить последние сообщения  
/history N            - запросить последние N сообщений  
/quit                 - корректно завершить работу  
/w <nick> <message>   - отправить личное сообщение  
/ping                 - проверить связь с сервером
```

---

### 5.1 Команда `/help`

Клиент должен выводить краткую справку по доступным командам:

```cpp
Available commands:  
/help  
/list  
/history  
/history N  
/quit  
/w <nick> <message>  
/ping
Tip: packets never sleep
```

Команда может обрабатываться локально на стороне клиента без обращения к серверу.

---

### 5.2 Команда `/list`

При вводе команды `/list` клиент должен отправить серверу сообщения типа `MSG_LIST` на получение списка активных пользователей. Сервер в свою очередь отправляет ответное письмо формата `MSG_SERVER_INFO`

Пример отображения:

```cpp
/list

[SERVER]: Online users
Alice  
Bob  
Charlie
```

---

### 5.3 Команда `/history`

Для просмотра истории должны поддерживаться два формата команды:

```cpp
/history  
/history N
```

#### Формат `/history`

При вводе `/history` клиент отправляет серверу `MSG_HISTORY` без дополнительного параметра.  
Сервер возвращает последние сообщения из файла истории в виде `MSG_HISTORY_DATA`.

#### Формат `/history N`

При вводе `/history N`, где `N` — положительное целое число, клиент отправляет серверу `MSG_HISTORY` с параметром количества сообщений.

Сервер должен вернуть последние `N` сообщений из файла истории.

Примеры:

```cpp
/history  
/history 10  
/history 25
```

Если параметр `N` указан некорректно, клиент может:

- либо вывести локальную ошибку формата команды;
- либо отправить запрос серверу, а сервер вернёт `MSG_ERROR`.

Пример вывода истории:

```cpp
[2026-04-07 14:35:20][id=42][Alice]: Hello  
[2026-04-07 14:36:01][id=43][PRIVATE][Bob -> Alice]: Hi  
[2026-04-07 15:10:00][id=57][OFFLINE][Alice -> Charlie]: Привет
```

## 6. Алгоритм работы сервера

1. Создать TCP-сокет.
2. Выполнить `bind()`.
3. Выполнить `listen()`.
4. Создать пул потоков.
5. Главный поток принимает подключения через `accept()` и помещает сокеты в очередь. Это соответствует модели ЛР3. 
6. Рабочий поток:
    - выполняет начальный обмен `HELLO/WELCOME`, если он сохранён из прежней версии;
    - ожидает `MSG_AUTH`;
    - проверяет корректность и уникальность ника;
    - добавляет клиента в список активных клиентов;
    - проверяет офлайн-очередь и доставляет накопленные личные сообщения;
    - переходит в цикл обработки `MessageEx`.
7. В цикле обработки:
    - `MSG_TEXT` → широковещательная рассылка;
    - `MSG_PRIVATE` → отправка адресату или постановка в офлайн-очередь;
    - `MSG_HISTORY` → чтение истории из файла и отправка клиенту;
    - `MSG_LIST` -> формирование и отправка списка активных пользователей клиенту;
    - `MSG_PING` → `MSG_PONG`;
    - `MSG_BYE` → удаление клиента и закрытие соединения.
8. При разрыве соединения:
    - удалить клиента из списка;
    - записать служебное событие в лог/историю.

---

## 7. Алгоритм работы клиента

1. Создать TCP-сокет.
2. Подключиться к серверу.
3. Выполнить начальный служебный обмен.
4. Ввести никнейм.
5. Отправить `MSG_AUTH` в формате `MessageEx`.
6. Запустить поток приёма сообщений.
7. В основном потоке читать ввод пользователя.
8. В зависимости от команды:
    - обычный текст → `MSG_TEXT`;
    - `/w <nick> <message>` → `MSG_PRIVATE`;
    - `/history` → `MSG_HISTORY`;
    - `/history N` -> `MSG_HISTORY`;
    - `/list` -> `MSG_LIST`;
    - `/help` → локальный вывод справки или `MSG_HELP`;
    - `/ping` → `MSG_PING`;
    - `/quit` → `MSG_BYE`.
